PREAMBLE — DESIGN PHILOSOPHY
V71 is an architecturally sound wave diagnostic engine with an underdeveloped decision layer. Engine 1A is perfect and untouched. The V72 mandate is integration, not reinvention: every proposed component reuses existing computed variables, elevating them from display artifacts into decision inputs.
The three laws of V72:
1. Engine 1A owns lifecycle. Nothing in V72 assigns, modifies, or competes with ie1a_currentPhase.
2. Every new engine consumes outputs — it does not duplicate computation.
3. Every variable must have a named downstream consumer. If it has no consumer, it does not exist.

2.2 Authority Conflict Prevention Rules
* Any variable that contains the word "Phase" in its name may only be assigned by IE1A.
* If two engines produce a variable of the same semantic type (e.g., a "narrative direction string"), they must have distinct names, distinct purposes, and the Dashboard must resolve which one surfaces in Standard Mode. Default rule: DOE output surfaces in Standard; all others in Advanced.
* The three competing directive strings (directiveStr, liveDirective, execDirective) in V71 are an authority conflict. In V72: DOE owns doe_action exclusively. liveDirective and execDirective are renamed to rie_controlDirective and mce_execReadiness and moved to Advanced Mode.

PART 3 — ROTATION INTELLIGENCE ENGINE (Engine 2 / RIE)
3.1 Purpose
Detect the probability that market control is transferring from one participant to another. This is NOT a lifecycle transition — it is a pressure gauge of emerging opposition. IE1A will detect the transition when it occurs (via BOS scoring); RIE detects the preconditions before they fully manifest.
3.2 Design
Inputs (all from existing V71 variables — no new computation):

Calculation Logic:

// rot_pressure: opposition gaining strength
float rot_pressure =
    math.min(
        obs_DecayScore * 0.35 +
        obs_AbsorptionScore * 0.30 +
        (convexityScore > 40 ? convexityScore * 0.20 : 0) +
        (obs_LiquidityScore > 50 ? 15 : 0),
        100.0)

// rot_controlStability: directional confidence
float rot_controlStability =
    math.min(
        obs_ExpansionScore * 0.40 +
        (efficiency > effThresh ? 30 : efficiency > effThresh * 0.7 ? 15 : 0) +
        (ede_state <= 2 ? 30 : ede_state <= 3 ? 15 : 0),
        100.0)

// rot_transferProbability: synthesis
float rot_transferProbability =
    math.min(
        rot_pressure * 0.50 +
        (100 - rot_controlStability) * 0.30 +
        (re_resolutionState == "UNRESOLVED" ? 20 : re_resolutionState == "PARTIALLY RESOLVED" ? 10 : 0),
        100.0)

// rot_emergingWaveStrength
float rot_emergingWaveStrength =
    math.min(
        obs_DecayScore * 0.40 +
        (convexityScore > 25 ? convexityScore * 0.30 : 0) +
        (not (bullImpulse or bearImpulse) and obs_AbsorptionScore > 40 ? 20 : 0),
        100.0)

// rot_state classification
string rot_state =
    rot_transferProbability >= 75 ? "TRANSFER_IMMINENT" :
    rot_transferProbability >= 50 ? "CONTESTED" :
    rot_transferProbability >= 25 ? "SOFTENING" : "STABLE"

PART 4 — MULTI-TIMEFRAME CONSENSUS ENGINE (MCE)
4.1 Timeframe Stack
V72 targets a 9-level stack. In PineScript, higher timeframes are accessed via request.security() with the existing tf1 / tf2 input mechanism. The trader configures the stack via inputs:

The current L1 (tf1, default 15m) and L2 (tf2, default 60m) map into two slots of the stack. The trader may reconfigure. The MCE treats the full stack uniformly.
4.2 Per-Timeframe Data Object
For each timeframe n, the MCE produces:

PART 9 — FRZ ↔ ERF UNIFICATION MODEL
9.1 Conceptual Relationship
FRZ and ERF describe the same energy phenomenon from different perspectives:
* ERF tracks energy that has been accumulated and must be delivered and resolved. It answers: Is the current wave's energy still unresolved?
* FRZ marks price zones where previous delivery was incomplete — zones to which price must return to complete resolution. It answers: Where does unresolved energy need to go?
Together: ERF tells you energy is outstanding. FRZ tells you where the market must travel to resolve it.
9.2 Formal Linkage Variables


pine
// frz_resolutionScore: how much of the FRZ's embedded energy has been resolved
// Calculated by cross-referencing FRZ age and RE resolution state
float frz_resolutionScore =
    re_resolutionState == "RESOLVED" ? 90.0 :
    re_resolutionState == "PARTIALLY RESOLVED" ?
        50.0 + re_recursiveCompletionScore * 0.40 :
    20.0 + ede_dissipationProgress * 0.30

// frz_residualEnergy: energy still embedded in the zone (inverse of resolution)
float frz_residualEnergy =
    math.max(0, 100.0 - frz_resolutionScore)

// frz_attractorWeight: how strongly this FRZ functions as a price attractor
// Zones with high residual energy attract price; fully resolved zones do not
float frz_attractorWeight =
    frz_residualEnergy * 0.50 +
    frz_bestScore * 0.30 +
    (frz_bestStatus == "Open" ? 20 : frz_bestStatus == "Partial" ? 10 : 0)

PART 10 — TARGET ENGINE (TE)
10.1 Purpose
Convert EAE attractors and FRZ zones into three named, prioritised, priced trade targets.
10.2 Target Hierarchy


TP1 ← Secondary Attractor (eae_secondaryAttractorPrice)
       OR nearest high-score FRZ above/below price
TP2 ← Primary Attractor (eae_primaryAttractorPrice)
TP3 ← Tertiary/Extended Attractor (eae_tertiaryAttractorPrice — new)
       OR furthest high-score FRZ
Rationale: The secondary attractor is the nearer, more likely resolution target; the primary is the full-energy destination; the tertiary is the maximum extension. This ordering is directionally consistent with energy theory: energy first partially resolves (TP1), then fully resolves (TP2), then extensions form (TP3).

PART 11 — INVALIDATION ENGINE (IE2)
11.1 Purpose
Pre-calculate the price levels at which each trade thesis fails. The trader sees the stop level before entering — not just an internal boolean that fires after invalidation.
11.2 Invalidation Types
Type	Trigger	Price Level
Bullish Origin Failure	Long thesis fails	point4OriginLow - ATR×0.25
Bearish Origin Failure	Short thesis fails	point4OriginHigh + ATR×0.25
Demand Failure	Demand Return invalidated	Demand zone bottom - ATR×0.10
Supply Failure	Supply Return invalidated	Supply zone top + ATR×0.10
11.3 Variables


pine
float inv_bullOriginPrice    // Stop level for long trades — origin low with buffer
float inv_bearOriginPrice    // Stop level for short trades — origin high with buffer
float inv_demandFailPrice    // Demand zone failure level
float inv_supplyFailPrice    // Supply zone failure level
float inv_activeStop         // Currently active stop (matches trade direction)
bool  inv_invalidated        // true when close has crossed inv_activeStop
float inv_riskInPts          // Distance from doe_entryZone.mid to inv_activeStop (absolute)
float inv_riskInATR          // inv_riskInPts / atr
11.4 Calculation


pine
float inv_bullOriginPrice =
    point4OriginLow - atr * 0.25

float inv_bearOriginPrice =
    point4OriginHigh + atr * 0.25

float inv_demandFailPrice =
    direction == 1 and closeInside ?
    (ie1a_currentPhase == "Demand Return" ?
     cycleHigh - (cycleHigh - cycleLow) * 0.60 - atr * 0.10 :  // approximate demand zone low
     inv_bullOriginPrice) : na

float inv_supplyFailPrice =
    direction == -1 and closeInside ?
    cycleHigh + atr * 0.10 : na   // approximate supply zone high

float inv_activeStop =
    direction == 1 ?
        math.min(inv_bullOriginPrice, nz(inv_demandFailPrice, inv_bullOriginPrice)) :
    direction == -1 ?
        math.max(inv_bearOriginPrice, nz(inv_supplyFailPrice, inv_bearOriginPrice)) : na

bool inv_invalidated = not na(inv_activeStop) and
    (direction == 1 ? close < inv_activeStop : close > inv_activeStop)

float inv_riskInPts = math.abs(close - inv_activeStop)
float inv_riskInATR = inv_riskInPts / atr
11.5 Risk-Reward Integration


pine
// R:R ratio for TP1, TP2, TP3
float te_rr_tp1 = math.abs(te_tp1 - close) / inv_riskInPts
float te_rr_tp2 = math.abs(te_tp2 - close) / inv_riskInPts
float te_rr_tp3 = math.abs(te_tp3 - close) / inv_riskInPts

// Gate: if R:R to TP1 < 1.5, degrade tqe_grade by one level
bool te_rrGate = te_rr_tp1 >= 1.5
